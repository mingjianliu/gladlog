# 全新重写路线图 + 子项目 0(自有代码合规审计)设计

日期:2026-07-10
状态:待用户审阅

## 背景与目标

上游 wowarenalogs 于 2025-12-08(commit `cf2e72ea`)将许可证从 MIT 改为 CC BY-NC-ND 4.0,禁止分发修改版。用户决定**彻底从零重写**一个属于自己的 WoW 竞技场日志分析产品:不使用任何上游代码(包括 MIT 期代码),只保留自己原创的代码资产。

**产品目标**:桌面应用安装包 + 开源代码仓库。本地优先:解析本地战斗日志、浏览战报、AI 复盘分析(直连 LLM API)。无登录、无云端上传、无社区对局浏览;第一版无录像功能。

## 已确认的设计决策

| 决策               | 选择                                              | 理由                                                                                                                                           |
| ------------------ | ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| 上游代码使用边界   | 一行不用(含 MIT 期)                               | 用户意愿:与原作者完全切割;附带收益:新项目无需保留任何第三方版权声明                                                                            |
| 技术栈             | Electron + React + TypeScript,Vite 取代 Next.js   | 与用户现有资产(React 组件、Node 工具链)同栈,迁移成本最低                                                                                       |
| 云端功能           | 全部砍掉                                          | 无服务器运维成本;用户已有本地 Next server 架构经验                                                                                             |
| 录像(recorder)     | 第一版不做                                        | 避开最大的一块重写工作量                                                                                                                       |
| 新仓库历史         | 全新仓库、orphan 单 commit 起步                   | 历史中永不出现上游代码;最简单                                                                                                                  |
| 游戏数据           | 自己写脚本从一手来源(暴雪 API / raidbots)重新生成 | 数据是暴雪的事实数据,可自由再导出;不带上游的 JSON 编译成果                                                                                     |
| 新 parser 数据模型 | 自由设计自己的模型 + **薄适配层**接下游           | 经 agy 辩论修正:全量兼容旧接口会把新 parser 反向锁死在上游设计上;改为 adapter 只映射下游实际消费的最小字段子集,下游代码是自己的,接口名可批量改 |

**合规原则**(贯穿所有子项目):

- 功能概念、日志格式知识、架构思路不受版权保护,可以参考;代码表达(expression)必须全新。
- 界面视觉要原创:功能概念(回合分段、死亡时间轴等)可参考,像素级模仿原版布局不行。
- "自己的代码"三分法:审计通过的文件 → 直接复制;含上游片段的 → 清洗片段;散落在上游文件里的自有修改 → 提取逻辑重新安家。
- 项目名、logo 不使用 "WoW Arena Logs" 及近似名。
- 现有 fork(含 ND 期代码)建议转 private,不再公开分发。

## 子项目 0:自有代码合规审计 — 详细设计

### 目的

在移植任何"自己的"文件之前,机器化排查它们是否内嵌上游代码,产出逐文件的处置清单。这是后续所有子项目的地基,也是开源后应对质疑的证据。

### 输入

- 自有新增文件清单:`git diff --diff-filter=A --name-only 7842b644(merge-base) main`(约 1347 个改动文件中的新增部分,~1000+ 个)
- 上游全量 blob 库:upstream/main 全历史的所有文件版本(MIT 期 + ND 期都算"上游",因为一行不用)

### 方法(三层检测,由粗到细)

1. **整文件精确匹配**:自有文件的 git blob hash 是否出现在上游任一 commit 中 → 出现即 `DERIVED`(整体来自上游)。
2. **复制起源检测**:`git log --follow -C -M` 检查文件是否以"复制/重命名上游文件"方式诞生 → 是则 `DERIVED`。
3. **片段级相似度**:把上游所有 TS/TSX/JS 文件的规范化行(去空白、去注释)建索引;对每个自有文件统计与任一上游文件的最长连续公共行块与共享行占比。阈值:连续 ≥ 8 行相同,或共享行占比 > 30% → 标记 `NEEDS_SCRUB` 并列出具体片段位置,人工复核。

工具为一次性脚本,放 `scratch/`(不进新仓库)。阈值可在跑完首轮后按误报率调整;调整须记录在报告中。

### 校准(防止工具自欺)

- **阳性对照**:取一个已知从上游复制的文件(如上游创建的 `generateDataManifest.ts` 之前的某个上游文件副本,或直接拿上游文件自测),工具必须报 `DERIVED`。
- **阴性对照**:取一个确定纯原创的文件(如某 eval 报告生成器),工具必须报 `CLEAN`。
- 两个对照不过,先修工具再跑全量。

### 输出

`docs/analysis/2026-07-XX-own-code-audit.md`,含:

- 逐文件三分类:`CLEAN`(直接复制)/ `NEEDS_SCRUB`(列出片段位置与来源文件)/ `DERIVED`(整体衍生,需从零重写)
- 附录:所有被用户修改过的上游文件(含 58 个"用户与 ND 期上游都改过"的重叠文件)中,自有修改 hunk 的提取清单(`git diff 7842b644..main -- <file>`),供子项目 4 提取逻辑时用
- 检测方法、阈值、对照结果记录

### 错误处理与边界

- 二进制/JSON 数据文件不做片段检测,单独按来源归属判断(已确认:`spellNames.json`、`talentModifiers.json`、`trinketItemIds.json` 为自有;`spellIdLists.json`、`spellClassMap.json` 为上游 ND 期,不带走)。
- 测试 fixture(真实战斗日志 .txt)是暴雪游戏输出,不是上游的版权物,可带走;但上游仓库自带的 fixture 若被复制进自有测试,标注来源为"上游仓库收集",建议换成自己采集的日志(用户有 windows-agent 采集管线,替换成本低)。
- 判定拿不准的文件一律降级处理(宁可 `NEEDS_SCRUB` 不放行)。

### 成功标准

- 全部自有新增文件有明确三分类结论,零"未审计"。
- 两个校准对照通过。
- `NEEDS_SCRUB`/`DERIVED` 文件各附具体证据(上游对应文件+行号),可人工抽查复核。

## 设计决策辩论记录(agy debate 仪式)

2026-07-10 对核心决策("彻底从零、一行不用")跑了 debate-open/reply(conversation `44891a10`,初始 OPPOSE → 最终 PARTIAL)。

**让步(已改设计)**:原方案"新 parser 数据模型兼容下游消费方式"自相矛盾——全量复刻上游几十种接口会把新 parser 架构反向锁死在上游设计上,且有衍生作品观感。已修正为**薄适配层**:新 parser 自由设计模型,adapter 只映射下游实际消费的最小字段子集。

**辩护成立**:未文档化的日志语义(如"免疫不触发 SPELL_AURA_APPLIED")是暴雪游戏输出的客观事实,不受版权保护;新 parser 用自采日志验证这些行为,不照上游代码抄。

**披露的隐性成本(用户须知情)**:AI 分析的经验阈值与 benchmark 校准数据是基于旧 parser 的输出拟合的。新 parser 在事件排序、容错、边缘判定上的细微差异会让这些校准漂移——子项目 4 不只是 import 替换,还包含一段**数据再对齐期**:阈值复核 + benchmark 重跑 + eval 体系回归。缓解手段:旧 fork 保留在本地作为**私有测试 oracle**(CC BY-NC-ND 不限制私人使用),对同一批 fixture 日志做新旧 parser 差分测试,量化漂移后再决定哪些阈值需要重拟合。
